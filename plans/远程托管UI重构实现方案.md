# WebHTV 远程托管 UI 重构实现方案

## 0. 本轮落地状态

已按本方案完成第一阶段 App 端 UI 重构：

- 远程托管入口拆成 `设备` 和 `设置` 两个主页面，设备动作进入独立详情页。
- 删除无效的 `打开服务地址` 浏览器入口。
- 新增 App 内 `检测服务`，请求 `{serverOrigin}/api/server/capabilities` 并解析展示 `serverMode/relayMode/capabilities/maxSyncPartBytes`。
- `绑定本机`、`添加设备`、`搜索/推送` 改为独立弹窗，避免首屏堆满输入框。
- 保留现有远程注册、绑定、刷新设备、状态、搜索、推送、最近日志协议。

暂不进入本轮的能力仍是完整 Web 控制台、远程文件管理、配置同步/恢复、批量设备管理和持久化服务端存储。

## 1. 背景问题

当前 `RemoteTrustDialog` 把这些任务全部挤在一个滚动页里：

- 中转服务配置。
- 本机注册和保持在线。
- 生成本机绑定码。
- 输入远端绑定码。
- 设备列表。
- 搜索、推送、状态、日志。
- 文件权限和清空托管。

这会带来三个直接问题：

- 小白用户无法判断当前 App 是主控端还是被控端。
- 高低频操作混在一起，添加设备、管理设备、设置服务互相干扰。
- `打开服务地址` 没有实际控制台页面，点击后只会看到空白或默认 Worker 响应，属于无效入口。

结论：不要继续在长表单里修补，应该按成熟远程设备工具的交互模式重构。

## 2. 调研结论

参考方向：

- KDE Connect：设备列表是主界面，配对是独立流程，设备动作在设备详情里。
- Syncthing：主界面按“文件夹/设备/状态”组织，同步状态可见，配置收进独立设置。
- RustDesk：默认展示可连接设备/地址，连接和控制是主任务，设置不占主界面。
- GitHub Mobile / Nextcloud / Amaze File Manager：列表页承载浏览，新增/上传/更多操作通过独立按钮或底部操作面板进入。
- Home Assistant：仪表盘先给状态总览，具体设备动作进入对应实体详情。
- Termius：主界面是主机列表，连接配置在独立编辑页，不让用户在首页面对所有字段。

可复用的最佳实践：

- 默认页先展示“状态和设备”，不要先展示大表单。
- 添加/绑定设备用独立短流程，不和服务设置混在一起。
- 设备动作放在设备详情或底部操作面板里。
- 日志和诊断按需打开，不常驻主界面。
- 设置项收进“设置”页，危险操作放到底部并二次确认。
- 无 Web 控制台时不要提供“打开服务地址”，应改成 App 内检测并解析服务能力。

## 3. 目标信息架构

入口仍然是：

```text
设置 -> 增强功能 -> 远程托管
```

进入后不是单页长表单，而是页面式面板：

```text
远程托管
├── 设备
│   ├── 本机状态
│   ├── 生成本机绑定码
│   ├── 已管理设备列表
│   └── 添加设备
├── 设备详情
│   ├── 状态
│   ├── 搜索
│   ├── 推送
│   └── 日志
└── 设置
    ├── 中转服务 URL
    ├── 检测服务
    ├── 保存/注册
    ├── 保持在线
    ├── 文件权限
    └── 清空托管
```

首屏只保留两类信息：

- 本机是否接入服务。
- 当前管理了哪些设备。

其他内容通过“添加设备”“生成绑定码”“设置”“设备详情”进入。

## 4. 页面设计

### 4.1 设备首页

顶部状态区：

```text
远程托管             [设置]
本机：在线 / 未接入 / 服务异常
服务：webhtv-remote...workers.dev
```

主动作：

```text
[生成本机绑定码]    [添加设备]
```

设备列表：

```text
客厅电视
在线 · 刚刚同步 · WebHTV 5.x

卧室盒子
离线 · 上次在线 06-17 21:30
```

空状态：

```text
暂无管理设备
在目标设备生成绑定码，然后点击“添加设备”。
```

### 4.2 添加设备

独立弹窗或子页面，只显示必要字段：

```text
添加设备

绑定码
[ 123456 ]

设备备注，可选
[ 客厅电视 ]

[添加]
```

成功后自动回到设备首页，并选中新设备。

### 4.3 生成本机绑定码

独立弹窗或子页面：

```text
本机绑定码

123456

10 分钟内有效

[复制] [刷新]
```

不要同时展示服务 URL、设备列表、推送输入框。

### 4.4 设备详情

手机端建议用底部操作面板或新页面；电视端建议用右侧详情区。

```text
客厅电视
在线 · dev_xxxxxxxx

[状态] [搜索] [推送] [日志]
```

点“搜索”后才出现关键词输入：

```text
搜索关键词
[ 流浪地球 ]
[发送搜索]
```

点“推送”后才出现 URL 输入：

```text
播放地址
[ https://... ]
[推送播放]
```

点“日志”后展示最近日志结果，并提供复制诊断信息。

### 4.5 设置页

设置页承载低频配置：

```text
中转服务
[ https://webhtv-remote.xiazy2011820123.workers.dev ]

[检测服务] [保存/注册]

服务状态：正常 / 未检测 / 连接失败
模式：simple
中转：origin-token-memory
能力：设备托管、远程同步、推送、最近日志
最大同步分片：25 MB

[保持远程托管在线]  开关
[文件权限]          未授权时显示

危险操作
[清空托管]
```

## 5. 去掉“打开服务地址”

`打开服务地址` 不应保留，原因：

- 简易 Worker 根路径没有 UI。
- 用户看到空白或 JSON 之外的页面会误以为部署失败。
- 真正有价值的是 `/api/server/capabilities` 返回的服务能力。

替代方案：

```text
检测服务
```

检测逻辑：

```text
GET {serverOrigin}/api/server/capabilities
```

例如：

```text
https://webhtv-remote.xiazy2011820123.workers.dev/api/server/capabilities
```

解析字段：

- `ok`
- `serverMode`
- `serverName`
- `relayMode`
- `time`
- `maxSyncPartBytes`
- `capabilities.configManage`
- `capabilities.remoteSync`
- `capabilities.pushAction`
- `capabilities.recentLog`
- 其他 capabilities 按需展示。

成功状态文案：

```text
服务可用
模式：simple
中转：origin-token-memory
支持：设备托管、远程同步、推送、最近日志
最大同步分片：25 MB
```

失败状态文案：

```text
服务不可用
原因：HTTP 404 / 网络超时 / JSON 格式异常
```

额外操作：

- `复制诊断信息`：复制 URL、HTTP 状态、响应 JSON 或错误信息。
- 不打开浏览器。

## 6. 手机端和电视端差异

手机端：

- 使用全屏 Dialog 或独立页面，不使用普通窄弹窗长滚动。
- 底部主导航或分段控件：`设备 / 设置`。
- 设备动作通过底部操作面板进入。
- 主按钮固定在底部或靠近设备列表顶部。

电视端：

- 使用左右分栏。
- 左侧：本机状态 + 设备列表。
- 右侧：选中设备详情和操作。
- 绑定码页和添加设备页要保证遥控器焦点顺序清晰。

## 7. 实现计划

### 阶段 1：服务检测和入口清理

- 删除 `打开服务地址` 按钮。
- `RemoteClient.capabilities()` 返回结构化 `ServerCapabilities` 模型。
- 新增“检测服务”动作。
- App 内解析 `/api/server/capabilities` 并展示服务状态。
- 增加“复制诊断信息”。

验收：

- 输入有效 Worker 地址后，点击“检测服务”显示 `simple/origin-token-memory` 和能力项。
- 输入错误地址时显示明确错误，不跳浏览器。

### 阶段 2：拆分主界面

- `RemoteTrustDialog` 从单页长表单改为页面式容器。
- 默认页只显示本机状态、主动作、设备列表。
- 服务 URL、保持在线、文件权限、清空托管移到设置页。
- 生成绑定码和添加设备改为独立子页面/弹窗。

验收：

- 首屏不出现搜索输入框、推送输入框、日志结果区。
- 用户能在 3 秒内理解本机状态和是否已有远端设备。

### 阶段 3：设备详情动作

- 设备列表点击后进入详情。
- `状态/搜索/推送/日志` 作为设备详情动作。
- 搜索和推送输入框只在对应动作打开后出现。
- 日志结果用独立结果面板展示，支持复制。

验收：

- 没有选中设备时不显示动作输入框。
- 远端设备离线时，命令结果显示“等待目标设备上线”。

### 阶段 4：视觉和焦点优化

- 设备列表使用紧凑列表项，不使用嵌套卡片。
- 在线状态用小 chip 或颜色点表达。
- 按钮文案短：`添加设备`、`绑定本机`、`检测服务`、`保存`。
- 手机端保证按钮不挤压、不截断。
- 电视端保证遥控器焦点顺序：设备列表 -> 详情动作 -> 设置。

验收：

- 手机小屏无文字重叠和按钮截断。
- 电视遥控器能完整走通注册、生成绑定码、添加设备、发送状态命令。

## 8. 建议文案

- `远程托管`：功能入口。
- `本机接入`：本机作为被控端接入服务。
- `绑定本机`：生成本机绑定码。
- `添加设备`：本机作为主控端添加远端设备。
- `检测服务`：请求 capabilities 并解析展示。
- `服务正常` / `服务异常` / `未检测`：顶部状态。
- `等待上线`：缓存设备但当前 Worker 尚未恢复在线路由。

避免使用：

- `打开控制台`：当前没有控制台。
- `打开服务地址`：没有实际用户价值。
- `管理远端`：概念偏抽象，建议主页面直接叫 `设备`。

## 9. 不进入本轮的内容

- 完整 Web 控制台。
- 远程文件管理。
- 远程一键同步 UI。
- 配置列表、配置切换、配置删除。
- 多设备批量操作。

这些内容应在设备基础页稳定后，以独立页面逐步加入。
