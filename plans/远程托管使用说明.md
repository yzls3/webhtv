# WebHTV 远程托管使用说明

## 1. 这个功能解决什么问题

远程托管的目标是让一台 WebHTV 设备可以被另一端远程管理，例如：

- 远程查看设备在线状态。
- 远程推送播放地址。
- 远程发起搜索。
- 拉取最近调试日志。
- 后续扩展远程同步、配置管理、文件管理等能力。

当前版本先落地 App 对等基础能力：同一台 App 既可以作为被控端注册、生成绑定码、保持在线轮询、接收基础命令，也可以作为主控端输入绑定码、保存设备组、刷新设备列表并下发基础命令。完整 Web 控制台和更完整的数据同步/文件管理仍属于后续阶段。

## 2. 先理解三个概念

- 中转服务 URL：你自己部署的远程 relay 地址，例如 `https://xxx.workers.dev`、`https://xxx.deno.dev`、`https://xxx.vercel.app`。
- 被控端：需要被远程管理的 WebHTV App，通常是电视、盒子或长期在线设备。
- 主控端：负责输入绑定码、保存设备组令牌、下发远程命令的一端。当前可以直接使用另一台 WebHTV App 的“管理设备”区域，后续也可以是 Web 控制台。

简易中转服务不保存长期账号和绑定关系。长期令牌保存在 App/主控端本地。只要中转服务域名不变，服务重启后通常不需要重新绑定。

## 3. 部署中转服务

仓库里目前提供三种零配置部署方式，选一种即可。

### 3.1 Cloudflare Worker

适合长期使用，推荐优先选择。

```bash
cd serverless/webhtv-remote-simple
npm install
cp wrangler.toml.example wrangler.toml
npm run deploy
```

部署完成后会得到类似：

```text
https://webhtv-remote-simple.xxx.workers.dev
```

这个地址就是 App 里要填写的“中转服务 URL”。

### 3.2 Deno Deploy

在 Deno Deploy 新建项目，入口选择：

```text
serverless/webhtv-remote-deno/main.js
```

部署完成后会得到类似：

```text
https://xxx.deno.dev
```

### 3.3 Vercel Edge

在 Vercel 新建项目时，把项目根目录设为：

```text
serverless/webhtv-remote-vercel
```

入口文件是：

```text
api/index.js
```

部署完成后会得到类似：

```text
https://xxx.vercel.app
```

## 4. 验证中转服务是否可用

优先在 App 内验证：

1. 打开 `设置 -> 增强功能 -> 远程托管 -> 设置`。
2. 填写中转服务 URL。
3. 点击 `检测服务`。

App 会请求：

```text
https://你的中转服务域名/api/server/capabilities
```

并解析展示服务模式、中转模式、支持能力和最大分片。能看到类似下面的返回，就表示中转服务可用：

```json
{
  "ok": true,
  "serverMode": "simple",
  "relayMode": "origin-token-memory"
}
```

如果检测失败，先检查部署平台的域名、路由和网络，不要先排查 App。

## 5. 在被控端 App 注册

在需要被远程管理的 WebHTV 设备上操作：

1. 打开 `设置 -> 增强功能 -> 远程托管`。
2. 进入 `设置` 页，在“中转服务 URL”里填写部署后的域名，例如：

```text
https://xxx.workers.dev
```

3. 保持“启用此服务”勾选。
4. 如果希望电视长期在线接收命令，勾选“保持远程托管在线”。
5. 点击“保存/注册”。

注册成功后，本机会生成并保存 `deviceToken`。后续 App 启动会自动向同一个中转服务恢复在线状态。

## 6. 生成绑定码

被控端注册后：

1. 回到 `设备` 页，点击“绑定本机”。
2. App 会显示 6 位绑定码。
3. 点击“复制”可以复制绑定码。
4. 绑定码短时间有效，过期后重新生成即可。

绑定码不是长期密码。它只是一次性把这台设备加入某个设备组的入口。

## 7. 主控端如何绑定设备

在作为主控端的另一台 WebHTV App 上操作：

1. 打开 `设置 -> 增强功能 -> 远程托管`。
2. 进入 `设置` 页，在“中转服务 URL”里填写同一个 relay 域名。
3. 点击“检测服务”，确认服务可用后点击“保存/注册”。
4. 回到 `设备` 页，点击“添加设备”，输入目标设备显示的 6 位绑定码。
5. 可选填写“设备备注”，例如 `客厅电视`。
6. 点击“添加设备”。

如果主控端本地还没有设备组，服务端会创建新的 `groupToken` 并返回给 App 保存；如果主控端已有设备组，App 会自动把新设备加入现有设备组。服务端同时给目标设备排队一个内部命令 `remote.profile.addGroup`，目标设备轮询到后会把该设备组写入本机信任配置。

绑定完成后，主控端会缓存设备列表；目标设备下一次注册或轮询后，点击“刷新设备”即可看到在线状态。

## 8. 界面按钮说明

- 中转服务 URL：填写你部署的 relay 域名。只认域名 origin，路径、参数不会作为绑定边界。
- 检测服务：请求 `/api/server/capabilities`，在 App 内显示服务模式、中转模式、支持能力和错误原因。
- 启用此服务：允许 App 使用这个中转服务注册和轮询。
- 保持远程托管在线：启动前台服务和保活锁，更适合电视、盒子等长期在线设备。
- 保存/注册：保存 URL，生成或复用本机设备令牌，并调用 `/api/device/register`。
- 绑定本机：生成短期绑定码，供另一台 WebHTV App 或 Web 控制台添加本机。
- 绑定码：主控端填写目标设备显示的 6 位短码。
- 设备备注：给目标设备设置一个容易识别的名称。
- 添加设备：调用 `/api/groups/claim`，保存设备组令牌，并尝试刷新设备列表。
- 刷新设备：调用 `/api/devices`，更新当前设备组里的在线设备缓存。
- 状态/搜索/推送/日志：对选中的远端设备下发 `device.status`、`action.search`、`action.push`、`log.recent`。
- 清空托管：删除本机保存的远程托管 profile、设备令牌、设备组和待绑定记录。

## 9. 当前版本支持的远程命令

当前 App 被控端已经支持执行：

- `device.status`：返回设备、App 版本、设备组数量、调试日志状态。
- `action.search`：在 App 内触发搜索。
- `action.push`：向 App 推送播放地址。
- `log.recent` / `device.log.recent`：返回最近调试日志。
- `remote.profile.addGroup`：绑定过程中由服务端下发，用于把被控端加入设备组。

当前 App 主控端已经支持：

- 输入绑定码添加远端设备。
- 缓存设备组和设备列表。
- 刷新远端设备列表和在线状态。
- 对选中设备下发状态、搜索、推送播放地址、最近日志命令。

还没有完整实现：

- 完整 Web 控制台。
- 配置列表、配置切换、配置删除。
- 远程一键同步导出/恢复。
- 远程文件管理。
- Git 云盘和远程托管的联动。

## 10. 数据保存和安全注意事项

App 会把远程托管资料保存到：

```text
WebHTV/RemoteTrust/profile.json
```

同时会在内部偏好里保存一份镜像。建议授予文件权限，这样卸载重装后仍可恢复托管关系。

普通一键同步和备份会排除：

```text
remote_trust_*
WebHTV/RemoteTrust/
```

这样可以避免把一台设备的 `deviceToken/groupToken` 同步到另一台设备，造成设备身份被克隆。

不要把 `profile.json`、`deviceToken`、`groupToken` 发给别人。拿到这些令牌的人可以控制对应设备组。

## 11. 常见问题

### 11.1 为什么填了 URL 后提示无效

需要填写完整可访问域名，例如：

```text
https://xxx.workers.dev
```

只填空字符串、错误协议或不可解析地址都会失败。

### 11.2 为什么不再提供“打开服务地址”

当前简易中转服务主要提供 API，不保证有图形化首页。根地址通常没有可操作控制台，所以 App 不再提供“打开服务地址”，改为 App 内请求：

```text
{中转服务 URL}/api/server/capabilities
```

并解析显示服务状态、模式、能力和错误原因，不再打开浏览器。

### 11.3 绑定码生成了，但设备没有加入组

绑定码只是第一步。主控端点击“添加设备”后，服务端会排队 `remote.profile.addGroup`，被控端需要保持 App 进程存活并完成下一次轮询。如果中转服务实例在 claim 后、被控端轮询前被平台回收，内部入组命令可能丢失，需要重新生成绑定码或由主控端补发 `remote.profile.addGroup`。

### 11.4 中转服务重启后要重新绑定吗

同一个域名不变时，一般不需要。App 会用本地保存的 `deviceToken/groupToken` 重新注册并恢复在线路由。

### 11.5 什么时候勾选“保持远程托管在线”

电视、盒子、常年供电设备建议勾选。手机、平板如果担心耗电，可以不勾选，只在 App 进程存活时接收远程命令。

## 12. 当前推荐使用流程

1. 部署 Cloudflare Worker 简易中转服务。
2. 在 App 的 `远程托管 -> 设置` 中填写中转服务 URL。
3. 点击“检测服务”，确认可用。
4. 在电视 WebHTV 打开 `设置 -> 增强功能 -> 远程托管`。
5. 进入 `设置` 页填写同一个中转服务 URL。
6. 勾选“启用此服务”。
7. 电视长期在线时勾选“保持远程托管在线”。
8. 点击“保存/注册”。
9. 回到 `设备` 页，点击“绑定本机”生成绑定码。
10. 在另一台 WebHTV App 打开同一个“远程托管”，填写相同中转服务 URL 并保存/注册。
11. 回到 `设备` 页，点击“添加设备”，输入绑定码和设备备注。
12. 点击“刷新设备”，选择目标设备。
13. 在设备详情页使用“状态/搜索/推送/日志”测试基础远程命令。

## 13. 当前 UI 结构

当前版本不再把配置、绑定码、设备列表、搜索、推送、日志全部挤在一个滚动页里，而是按任务拆成：

- `设备`：默认页，只展示本机状态、绑定本机、添加设备、设备列表。
- `设备详情`：选中设备后再展示状态、搜索、推送、日志。
- `设置`：中转服务 URL、检测服务、保存/注册、保持在线、文件权限、清空托管。

`检测服务` 请求：

```text
GET {serverOrigin}/api/server/capabilities
```

成功后展示：

```text
服务可用
模式：simple
中转：origin-token-memory
支持：设备托管、远程同步、推送、最近日志
最大同步分片：25 MB
```

失败时展示明确错误原因，并提供 `复制诊断信息`。
